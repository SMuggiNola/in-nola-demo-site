# BUILD SPEC: "The Tenement" — IN-NOLA Interactive Literary Experience

## Overview

An interactive literary experience where IN-NOLA members "visit" the Boyle family flat from Sean O'Casey's *Juno and the Paycock*. The member arrives as a neighbour with a personal problem. The characters of the play respond — but they're caught up in their own crises, and the member's problem is progressively forgotten as the play's three-act arc unfolds around them.

The experience runs in three scenes. Each scene uses a fresh API call to Mistral with an act-specific system prompt. The UI presents the dialogue as a group chat with typing indicators, staggered message reveals, and periodic prompts where the member can respond.

## Architecture

### Stack
- **Frontend:** Static HTML/CSS/JS hosted on Cloudflare Pages (mobile-first)
- **Backend:** Cloudflare Workers for API orchestration, session management, member auth
- **Storage:** Cloudflare Workers KV for member profiles, session history, donation tracking
- **Inference:** Mistral API (not local hardware — this needs to be reliable for events)
- **Auth:** Simple member login — IN-NOLA membership list, stored in KV. No OAuth. Username + a shared passphrase, or a unique member code.

### API Flow
```
Member opens app → Auth check against KV
  → Member types their problem
  → Worker builds system prompt (Scene 1 prompt + member's problem)
  → Worker calls Mistral API
  → Full dialogue response returned
  → UI parses and performs the dialogue as a group chat
  → At MEMBER_PROMPT points, UI pauses and shows response options
  → Member selects response
  → Worker builds Scene 2 prompt (includes member's problem + their Scene 1 choice)
  → Mistral API call
  → UI performs Scene 2
  → Repeat for Scene 3
  → Session ends → Summary generated → Stored in KV
```

### Mistral API Details
- Use Mistral's chat completions endpoint
- Model: Use whatever model Sean is currently running for his teacher pipeline (likely Mistral Medium or Mistral Large)
- Temperature: 0.7–0.8 (needs variance for replayability)
- Max tokens: 1500 per scene
- System prompt: The scene-specific markdown file (scene1-act1-prompt.md, scene2-act2-prompt.md, scene3-act3-prompt.md)
- User message: The member's problem + any previous responses

## Frontend Design

### Visual Identity
- **Aesthetic:** Dublin tenement, 1920s. Warm, worn, intimate. Think sepia tones, aged paper textures, gas lamp warmth.
- **Typography:** Serif font (Georgia, Playfair Display, or similar). The text should feel literary, not techy.
- **Mobile-first:** This will primarily be used on phones at events.

### Screen Flow

#### 1. Entry Screen
- IN-NOLA logo
- "The Tenement" title
- "Juno and the Paycock — An Interactive Experience"
- Member login (code or username)
- Brief flavour text: *"You're a neighbour in a Dublin tenement, 1922. The Boyle family lives on your landing. You've got a problem and you need advice. Knock on the door."*

#### 2. Problem Entry Screen
- A door illustration or simple UI element
- Text input: "What's happened? What do you need help with?"
- Placeholder text: *"I said something terrible at dinner and now my sister won't talk to me..."*
- Submit button: "Knock"
- Keep it to one text field. No categories, no dropdowns. Just tell us your problem.

#### 3. Chat Screen (Used for all 3 scenes)
- Group chat UI. Messages appear one at a time with typing indicators.
- Each character gets a distinct colour/avatar:
  - **Boyle:** Deep red/maroon
  - **Joxer:** Olive/khaki
  - **Juno:** Navy blue
  - **Johnny:** Pale grey
  - **Mary:** Soft green
  - **Mrs. Madigan:** Purple (Scene 2 only)
- Character name appears above each message bubble
- **Typing indicator:** Three dots animation, with the character's name: "Boyle is typing..."
- **Member prompts:** When the dialogue reaches a MEMBER_PROMPT point, the chat pauses and the member gets 2-3 response options as tappable buttons. These are generated by the AI as part of the dialogue output. Format in the API response:
  ```
  MEMBER_PROMPT: ["Option A text", "Option B text", "Option C text"]
  ```
- **Scene transitions:** Between scenes, a brief interstitial: *"A few days pass..."* or *"Two months later. November."* with the lighting/mood description from the system prompt. Fade to black, fade back in.
- **No timer visible.** The experience is paced by the dialogue, not a clock.

#### 4. End Screen
- After Scene 3 completes, fade to black.
- A moment of silence.
- Then Juno's final line appears alone on screen, centered, in larger text.
- Below it, small: *"Juno and the Paycock was first performed at the Abbey Theatre, Dublin, 1924."*
- Then: *"This experience was brought to you by Irish Network New Orleans."*
- Donation prompt (in character — rotate which character delivers it):
  - Boyle: *"If you've enjoyed the hospitality of this house — an' I use the word loosely — you might leave a small contribution. I'd do it meself, but I've a terrible pain in me legs."*
  - Joxer: *"Ah, that's a darlin' experience, a daarlin' experience. If you've a coin to spare, it'd be a darlin' thing entirely."*
  - Juno: *"If you got somethin' out of this, leave what you can. God knows we could all use a little more kindness."*
- PayPal button: IN-NOLA's PayPal donation link. $0.99 "cup of coffee."
- **Donation tracking:** Store `asked: true` and `donated: true` flags in KV. Only show the donation prompt once per member. Ever. If they've been asked before, skip straight to "Thank you for visiting the Boyles."

#### 5. Return Visit Handling
- If a member returns, the system prompt for Scene 1 gets an additional line injected from their stored session summary:
  - *"This neighbour has visited before. Last time they came by about [brief summary]. Boyle vaguely remembers them. Joxer claims they're old friends. Juno recalls the detail. Johnny doesn't care."*
- This is stored in KV as a 2-3 sentence summary generated by one final Mistral call at the end of each session.

## UI Performance / Chat Playback Engine

This is the critical piece. The AI returns the full dialogue as a text block. The UI must parse it and perform it as a live group chat.

### Parsing
The API returns text in the format:
```
CHARACTER_NAME: "Dialogue text here."
(Stage direction in parentheses.)
MEMBER_PROMPT: ["Option A", "Option B"]
```

The parser should:
1. Split by character lines (regex: `/^([A-Z\s."]+):\s*"(.+)"$/gm`)
2. Identify stage directions (lines starting with `(`)
3. Identify MEMBER_PROMPT lines and extract the options array

### Playback Timing
For each message:
1. **Reading delay:** After the previous message appears, wait 2-4 seconds (randomized) to simulate other characters "reading" it
2. **Typing indicator:** Show "[Character] is typing..." for a duration proportional to message length:
   - Short message (< 30 words): 1.5-2.5 seconds
   - Medium message (30-80 words): 3-5 seconds
   - Long message (80+ words, mainly Boyle's rants): 5-8 seconds
3. **Message reveal:** Message appears. Typing indicator for next character begins after the reading delay.
4. **Stage directions:** Appear as italicized system messages (grey, centered) with a brief pause.
5. **MEMBER_PROMPT:** Chat pauses completely. Response buttons appear at bottom. Timer: member has 30 seconds to choose or the system picks the first option. After selection, chat resumes.

### Multiple characters "typing"
Occasionally (2-3 times per scene), show two characters typing simultaneously, then one sends first and the other's dots disappear briefly before reappearing — as if they rewrote their message after seeing what the other person said. This sells the illusion of a real group chat.

## Session Management

### KV Schema
```
member:{code} → {
  name: "Sean",
  sessions: [
    {
      date: "2026-03-08",
      problem: "I said something terrible at dinner...",
      summary: "The neighbour came about a family argument. Boyle compared it to the Treaty negotiations. Juno told them to apologize before pride eats them alive.",
      donated: false
    }
  ],
  asked_donation: true
}
```

### Concurrency
- No hard limit on concurrent users — Mistral API handles this
- Each session is 3 API calls, each returning in seconds
- The UI playback takes minutes, so API load is minimal even with 15 members playing simultaneously

### Admin
- A simple admin endpoint: `yourdomain.com/admin?key=SECRET&action=stats` returns active sessions, total sessions, donation count
- `?action=pause` puts the experience in maintenance mode
- `?action=resume` brings it back

## System Prompt Integration

Three system prompt files are provided:
- `scene1-act1-prompt.md` — Hope and daily chaos
- `scene2-act2-prompt.md` — The party and the funeral
- `scene3-act3-prompt.md` — Everything collapses, Juno's truth

For each scene, the Worker:
1. Loads the scene's system prompt markdown
2. Appends the member's problem as a user message
3. For Scenes 2 and 3, includes a brief recap: "The neighbour previously said: [their Scene 1 and/or Scene 2 responses]"
4. Calls Mistral API with system prompt + user message
5. Returns the dialogue block to the frontend

### Injecting the Member

The user message sent to Mistral should be formatted as:

**Scene 1:**
```
The neighbour's name is {member_name}. They knock on the door and say:

"{member_problem}"

Generate the scene dialogue. Include exactly 2 MEMBER_PROMPT points where the neighbour can respond. Format member prompts as: MEMBER_PROMPT: ["option 1", "option 2", "option 3"]
```

**Scene 2:**
```
The neighbour {member_name} has returned. Their original problem was: "{member_problem}"

In the previous visit, when asked, they said: "{scene1_choice}"

Generate the scene dialogue. The characters vaguely remember the neighbour but are too caught up in the party to focus on their problem. Include exactly 2 MEMBER_PROMPT points.
```

**Scene 3:**
```
The neighbour {member_name} has returned one last time. Their original problem was: "{member_problem}"

Previously they said: "{scene1_choice}" and "{scene2_choice}"

Generate the final scene. The family is collapsing. The neighbour's problem is overwhelmed by catastrophe. But at the end, Juno — speaking from her own grief — must say one thing that accidentally, perfectly addresses the neighbour's original problem. She doesn't know she's helping them. She's just telling the truth.

Include exactly 2 MEMBER_PROMPT points.
```

### Post-Session Summary
After Scene 3 completes, make one final Mistral call:
```
System: You are a concise summarizer. Write a 2-3 sentence summary of this neighbour's visit to the Boyle flat, written from the perspective of someone who lives in the tenement. Use Dublin dialect.

User: The neighbour {member_name} came about this problem: "{member_problem}". They said: "{scene1_choice}" and "{scene2_choice}" and "{scene3_choice}". Juno's final words to them were: "{juno_final_line}".
```

Store this summary in KV for future visit injection.

## IN-NOLA Website Integration

### Site Sections

This experience lives within the existing IN-NOLA website (Cloudflare Pages). The site has several sections. The two relevant ones:

#### The Library
The Library is the IN-NOLA members' reading room. This is where the full text of *Juno and the Paycock* lives — formatted as a clean, readable, mobile-friendly presentation of the play. Not a PDF dump. A proper digital edition:

- **Clean typography:** Georgia or a good serif. Readable on mobile.
- **Character names bolded** with consistent styling
- **Stage directions in italics**, slightly indented or in a lighter colour
- **Act breaks** as clear section dividers with the setting description styled as scene-setting prose (not stage notation)
- **Navigation:** Jump to Act I / Act II / Act III
- **Context notes:** Brief, unobtrusive footnotes or margin notes for references an American reader might not know (Parnell, the Fenians, Easter Week, the Free State, Die-hards, etc.)
- The Library can host other works over time — this is the first. The O'Casey play is the inaugural text. Future additions could include other Dublin Trilogy plays, Yeats, Synge, Behan, etc.

#### The Village
The Village already exists as the IN-NOLA members-only interactive space. "The Tenement" is simply a new house in the Village — another location members can visit.

- The Tenement appears as a new location card/link within the existing Village navigation
- The Tenement entrance is the door/login screen described in the Frontend Design section above
- Future literary experiences (other plays, other worlds) become additional houses in the Village

The relationship: **The Library is where you read the play. The Village is where you live in it.**

### Landing Page Carousel

The IN-NOLA landing page features a **carousel** of announcement cards. Currently there is a card for the 64 Parishes article about J.F.X. O'Brien (the Irish Republican in Louisiana). A second card announces The Tenement experience.

#### Carousel Spec
- **Format:** Horizontal swipe carousel, mobile-friendly, auto-advances every 6 seconds with manual swipe/arrow override
- **Cards:** Each card is a full-width image or styled panel with headline, brief description, and CTA button
- **Minimum 2 cards at launch**, designed to grow as IN-NOLA adds content

#### Card 1: J.F.X. O'Brien Article (existing)
- Headline: The story of an Irish Republican in Louisiana (or similar existing copy)
- CTA: "Read on 64 Parishes" — links to the Andrew article
- Visual: Period-appropriate styling, Irish/Louisiana visual elements

#### Card 2: The Tenement Launch
- Headline: **"The Tenement — Step Inside Juno and the Paycock"**
- Subhead: *"An interactive literary experience. Visit the Boyle family flat. Bring your problems. They have enough of their own."*
- CTA: "Enter the Village" — links to the Village / Tenement login (members only)
- Visual: Dublin tenement aesthetic — warm sepia tones, aged walls, the glow of a votive light. Could use a simple illustration of a tenement door with a knocker, or a stylized version of the flat's layout from the play's stage directions.
- Badge: "Members Only" — subtle indicator that this is a membership benefit
- Secondary link: "Not a member? Join IN-NOLA" — membership page

#### Carousel Technical Notes
- CSS scroll-snap for smooth mobile swiping
- Dot indicators showing current card position
- Auto-advance pauses on touch/hover
- Cards should be easily addable — just an array of card objects with headline, subhead, image, CTA text, CTA link, and optional badge
- Keep the carousel component reusable for future cards (Village Tribune issues, event announcements, new Library additions, etc.)

## What Success Looks Like

A member at an IN-NOLA event pulls out their phone, logs in, types "I accidentally told my mother-in-law her cooking was terrible," and watches a family in a Dublin tenement argue about it while their own lives fall apart. They laugh in Scene 1. They feel uneasy in Scene 2. They're quiet in Scene 3. And when Juno says the thing — the one true thing — they put their phone down for a moment before they tap the coffee button.

That's the product.
